© All in Books 2026  •  Confidential  Page 2 1. Architectural Overview  
All in Books is built as a single -page React application packaged as a Progressive Web App. The 
architecture is deliberately simple: there is no backend server, no database server, and no API of the 
developer's own. Every piece of user data lives on the us er's device inside IndexedDB, managed 
through Dexie.js. Google Drive is the only external service touched, and only for backup — it is never 
on the critical path for any core operation.  
 
1.1 The Three -Layer Model  
The entire application maps onto three layers:  
 
┌─────────────────────────────────────────────────────────┐  
│  PRESENTATION LAYER                                      │  
│  React components + Tailwind CSS                         │  
│  Pages: Dashboard, Entries, Parties, Invoices, Reports  │  
│  Forms: EntryForm, InvoiceForm, ImportFlow, Settings     │  
└────────────────────┬────────────────────────────────────┘  
                     │  reads/writes via Zustand stores  
┌────────────────────▼────────────────────────────────────┐  
│  DATA ACCESS LAYER                                       │  
│  Zustand stores (transactionStore, partyStore, etc.)    │  
│  Each store wraps Dexie.js table operations             │  
│  Exposes: add(), update(), delete(), query(), getAll()  │  
└────────────────────┬────────────────────────────────────┘  
                     │  reads/writes  
┌────────────────────▼────────────────────────────────────┐  
│  PERSISTENCE LAYER                                       │  
│  Dexie.js  →  IndexedDB (device storage)                │  
│  db.transactions / db.parties / db.invoices             │  
│  db.categories / db.books / db.recurringEntries         │  
│  db.importBatches / db.attachments (blob store)         │  
└─────────────────────────────────────────────────────────┘  
 
Separately, two utility modules handle outbound operations and never touch the Zustand stores 
directly:  
• ExportService — reads Dexie tables, uses SheetJS to build the .xlsx export in memory, triggers a 
browser download  
• BackupService — calls ExportService to get the xlsx buffer, then uploads it to Google Drive via the 
REST API

© All in Books 2026  •  Confidential  Page 7 3. Database Layer — Dexie.js Schema  
The Dexie instance is defined once in src/db/db.js and imported everywhere that needs data access. 
Dexie uses schema versioning to handle migrations — when the schema changes, increment the 
version number and provide an upgrade function. The full v1 schema  is defined below.  
 
// src/db/db.js  
import Dexie from 'dexie';  
 
export const db = new Dexie('AllInBooks');  
 
db.version(1).stores({  
  // Indexed fields only (Dexie indexes, not all columns)  
  // All non -indexed fields are stored automatically  
  transactions:  '++id, bookId, type, date, partyId, categoryId, 
isImported, importBatchId',  
  parties:       '++id, bookId, name',  
  invoices:      '++id, bookId, partyId, status, invoiceNumber',  
  categories:    '++id, bookId, type',  
  books:         '++id',  
  importBatches: '++id, bookId, importedAt',  
  recurringEntries: '++id, bookId, nextDueDate, isActive',  
  attachments:   '++id, transactionId',   // blob store  
}); 
 
// Schema upgrade pattern:  
// db.version(2).stores({ ... }).upgrade(tx => { ... });  
 
3.1 Index Strategy  
Only fields that will be used in .where() or .orderBy() queries are listed in the Dexie schema string. All 
other fields are stored as part of the object but are not indexed. This keeps the index size small and 
writes fast.

© All in Books 2026  •  Confidential  Page 9 4. State Management — Zustand Stores  
Each domain entity has its own Zustand store. Stores are the only point of contact between React 
components and the Dexie database. Components never import db directly — they call store actions. 
This makes the data layer swappable and testable independentl y of UI.  
 
4.1 Store Pattern — transactionStore  
Every store follows the same pattern: a slice of in -memory state, async actions that write to Dexie and 
update state, and query helpers that read from Dexie on demand.  
// src/stores/transactionStore.js  
import { create } from 'zustand';  
import { db } from '../db/db';  
import { generateId } from '../utils/uuid';  
 
export const useTransactionStore = create((set, get) => ({  
  // State  
  transactions: [],          // currently loaded page/filter  
  loading: false,  
 
  // Load transactions for a filter (type, dateRange, partyId)  
  load: async ({ type, from, to, partyId } = {}) => {  
    set({ loading: true });  
    let query = db.transactions.orderBy('date').reverse();  
    if (type)    query = query.filter(t => t.type === type);  
    if (partyId) query = query.filter(t => t.partyId === partyId);  
    if (from)    query = query.filter(t => t.date >= from);  
    if (to)      query = query.filter(t => t.date <= to);  
    const transactions = await query.toArray();  
    set({ transactions, loading: false });  
  }, 
 
  // Add a new transaction  
  add: async (data) => {

    await db.transactions.update(id, { ...changes, updatedAt: Date.now() 
}); 
    get().load();  
  },

4.3 getTodaySummary — Dashboard Query  
This is the most frequently called query (runs on every dashboard render). It must be fast. It reads all 
transactions for today's date from IndexedDB, sums them into a {totalIn, totalOut, net} object, and also 
computes the total outstanding receivables and  payables across all parties.  
// Inside transactionStore  
getTodaySummary: async () => {  
  const today = new Date().toISOString().split('T')[0];  // 'YYYY -MM-DD' 
  const todayTx = await db.transactions  
    .where('date').equals(today).toArray();  
 
  const totalIn  = todayTx.filter(t => 
['sale','receipt'].includes(t.type))  
                          .reduce((s, t) => s + t.amount, 0);  
  const totalOut = todayTx.filter(t => 
['purchase','expense','payment'].includes(t.type))  
                          .reduce((s, t) => s + t.amount, 0);  
 
  return { totalIn, totalOut, net: totalIn - totalOut };  
},

Service modules are plain JavaScript classes or objects (no React, no Zustand). They are called from 
React components or store actions. They have a single responsibility each and return Promises.  
 
5.1 ExportService  
Reads all Dexie tables for the active book and builds a multi -sheet .xlsx file in memory using SheetJS. 
Returns an ArrayBuffer that can be downloaded or passed to BackupService.  
// src/services/ExportService.js  
import * as XLSX from 'xlsx';  
import { db } from '../db/db';  
 
export const ExportService = {  
  async generate(bookId = 'main') {  
    const wb = XLSX.utils.book_new();  
 
    // Sheet 1: Transactions  
    const txs = await 
db.transactions.where('bookId').equals(bookId).toArray();  
    XLSX.utils.book_append_sheet(wb, XLSX.utils.json_to_sheet(txs), 
'Transactions');  
 
    // Sheet 2: Parties (with computed balance)  
    const parties = await 
db.parties.where('bookId').equals(bookId).toArray();  
    // ... compute balance per party from transactions ...  
    XLSX.utils.book_append_sheet(wb, XLSX.utils.json_to_sheet(parties), 
'Parties');  
 
    // ... repeat for Invoices, Categories, GST_Summary, Monthly_Summary, 
Import_Log  
 
    return XLSX.write(wb, { type: 'array', bookType: 'xlsx' });  
  }, 
 
  downloadNow(buffer, filename) {  
    const blob = new Blob([buffer], { type: 'application/octet -stream' });  
    const url  = URL.createObjectURL(blob);  
    const a    = document.createElement('a');  
    a.href = url; a.download = filename; a.click();  
    URL.revokeObjectURL(url);  
  } 
};

export const BackupService = {  
  async run(bookId = 'main') {  
    const token  = await DriveAuth.getToken();       // refresh if expired  
    const buffer = await ExportService.generate(bookId);  
    const folder = await this._getOrCreateFolder(token);

    const now  = Date.now();  
    const day  = new Date().getDay();  // 0 = Sunday  
    return day === 0 && (now - last) > 6 * 24 * 60 * 60 * 1000;  
  }

// src/utils/gst.js  
 
export const GST_RATES = [0, 5, 12, 18, 28];  
 
// User enters total (inclusive). Calculate base and GST.  
export function calcFromTotal(total, rate) {  
  if (!rate || rate === 0) return { baseAmount: total, gstAmount: 0 };  
  const base = parseFloat((total / (1 + rate / 100)).toFixed(2));  
  const gst  = parseFloat((total - base).toFixed(2));  
  return { baseAmount: base, gstAmount: gst };  
} 
 
// User enters base. Calculate GST and total.  
export function calcFromBase(base, rate) {  
  if (!rate || rate === 0) return { gstAmount: 0, total: base };  
  const gst   = parseFloat((base * rate / 100).toFixed(2));  
  const total = parseFloat((base + gst).toFixed(2));  
  return { gstAmount: gst, total };  
} 
 
// Validate that base + gst ≈ total (within ₹1 tolerance)  
export function validateGST(total, base, gst) {  
  return Math.abs(total - base - gst) <= 1;  
}

// src/hooks/usePWAInstall.js  
import { useState, useEffect } from 'react';  
 
export function usePWAInstall() {  
  const [prompt, setPrompt] = useState(null);  
 
  useEffect(() => {  
    const handler = (e) => { e.preventDefault(); setPrompt(e); };  
    window.addEventListener('beforeinstallprompt', handler);  
    return () => window.removeEventListener('beforeinstallprompt', 
handler);  
  }, []);  
 
  const promptInstall = async () => {  
    if (!prompt) return;  
    prompt.prompt();  
    await prompt.userChoice;  
    setPrompt(null);  
  }; 
 
  return { canInstall: !!prompt, promptInstall };  
}

function downloadTemplate() {  
  const binary = atob(IMPORT_TEMPLATE_B64);  
  const bytes  = new Uint8Array(binary.length).map((_,i) => 
binary.charCodeAt(i));  
  const blob   = new Blob([bytes], { type: 'application/octet -stream' });  
  const url    = URL.createObjectURL(blob);  
  const a      = document.createElement('a');  
  a.href = url;  
  a.download = 'AllInBooks_ImportTemplate.xlsx';  
  a.click();  
  URL.revokeObjectURL(url);  
}

The app uses the OAuth 2.0 Authorization Code flow with PKCE (Proof Key for Code Exchange) — the 
correct flow for a client -side app with no backend. Tokens are stored in localStorage (access token with 
expiry, refresh token). The app never sees the user's Google password.  
 
10.1 Setup (One -Time Developer Steps)  
1. Create a project in Google Cloud Console  
2. Enable the Google Drive API  
3. Create an OAuth 2.0 Client ID of type 'Web application'  
4. Add the Vercel deployment URL to Authorised JavaScript Origins  
5. Add the Vercel deployment URL + /auth/callback to Authorised Redirect URIs  
6. Set the OAuth consent screen: app name 'All in Books', scope: 
https://www.googleapis.com/auth/drive.file  
7. Copy the Client ID into src/services/DriveAuth.js as GOOGLE_CLIENT_ID

14.1 Environment Variables  
The app has exactly two environment variables. Both are public (they are embedded in the client -side 
bundle). Neither is a secret — the Google Client ID is a public identifier, not a credential.  
# .env.local (development)  
VITE_GOOGLE_CLIENT_ID=your -client-id.apps.googleusercontent.com  
VITE_APP_VERSION=1.0.0  
 
# .env.production (Vercel environment variables)  
VITE_GOOGLE_CLIENT_ID=your -production -client-id.apps.googleusercontent.com  
VITE_APP_VERSION=1.0.0  
  NOTE  There is no .env.secret, no API key, no database connection string. The app has no secrets to protect. 
Google OAuth security is provided by PKCE and the registered redirect URI, not by a client secret.  
 
14.2 Vercel Deployment  
The app deploys to Vercel from the main branch of the GitHub repository. No special build configuration 
is required beyond the standard Vite setup.  
# Vercel build settings (auto -detected from vite.config.js)

Output Dir:      dist  
Install Command: npm install
